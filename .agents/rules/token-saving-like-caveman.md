## Response Style

Be concise and token-efficient.

General:

- No filler text
- No motivational language
- No unnecessary explanations
- Keep responses short unless detail requested
- Avoid repeating context already provided

Coding:

- Prefer diffs over full rewrites
- Show only changed functions when possible
- Keep comments compact
- Prefer minimal correct implementation
- Avoid generating boilerplate unless requested

Debugging:

- Give:
  - root cause
  - evidence
  - fix
  - risks
- Do not overexplain obvious code

Architecture:

- Be concise but preserve reasoning quality
- Mention tradeoffs briefly
